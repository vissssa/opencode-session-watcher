3. 模块分析

3.1 Domain 层 — internal/domain/domain.go

项目的核心抽象层，定义全部领域类型和接口契约。这是整个系统的"合约中心"。

核心类型

类型职责关键字段
Session AI 对话会话元数据 ID, UserID, AgentID, UpdatedAt, Raw json.RawMessage
Message 会话消息元数据 ID, SessionID, CreatedAt, Raw json.RawMessage
MessageRecord 写入 Sink 的完整记录（envelope） JSON 字段 + 输出追踪字段（json:"-"）
设计决策：Raw JSON 透传 Session.Raw 和 Message.Raw 保存完整原始 JSON，避免因 API 字段新增导致信息丢失。这是一种"透明代理"模式：系统不解析业务语义，只负责数据搬运，保持最大向前兼容性。

4.3 增量边界探测算法（fetchUntilBoundary）

这是系统中最核心的算法，解决"如何在不知道上次同步位置的情况下，确保不遗漏新消息"的问题。

flowchart TD Start([开始]) --> Init["limit = min(step, maxFetch)"] Init --> Fetch["GET /message?limit=N\n返回最近 N 条消息"] Fetch --> CheckExists{"包含已处理消息?"} CheckExists -->|Yes| Found["找到边界，停止扩展\nreachedMax=false"] CheckExists -->|No| CheckCount{"返回数 < limit?"} CheckCount -->|Yes| AllFetched["已取完全部消息\nreachedMax=false"] CheckCount -->|No| CheckMax{"limit >= maxFetch?"} CheckMax -->|Yes| Overflow["触顶！warn 日志\nreachedMax=true"] CheckMax -->|No| Expand["limit = min(limit+step, maxFetch)\n扩大窗口"] Expand --> Fetch Found --> Filter["过滤出未处理消息\n按 CreatedAt+ID 排序"] AllFetched --> Filter Overflow --> Filter Filter --> End([返回消息列表]) style Overflow fill:#450a0a,color:#ef4444 style Found fill:#14532d,color:#22c55e style AllFetched fill:#14532d,color:#22c55e
算法关键假设 API 的 limit 语义必须是"返回最近 N 条"（按时间倒序截取），否则边界探测失效。若 open-code 服务修改了消息排序语义，此算法需要同步调整。

4.4 双重防重写入流程

PrepareMessageRecords 采用三步防重策略，确保并发场景下不丢失也不重复写入：

展开关键 SQL 逻辑 
-- 步骤 1: 不存在则插入 pending（并发安全）
INSERT OR IGNORE INTO messages(id, status, ...) VALUES (?, 'pending', ...)

-- 步骤 2: 存在且非 written 时，更新元数据并重置为 pending（幂等重试）
UPDATE messages SET status='pending', ...
WHERE id = ? AND status <> 'written'

-- 步骤 3: 查询最终状态，跳过已 written 的记录
SELECT status FROM messages WHERE id = ?
-- 只有 status != 'written' 的记录才加入待输出列表
Section 5

5. 关键源码解析

5.1 shouldSync — Session 是否需要同步

展开源码 
func shouldSync(session domain.Session, state store.SessionState, found bool) bool {
    if !found {
        return true // 首次见到该 Session
    }
    if session.UpdatedAt == 0 {
        return true // 远端无更新时间，保守处理
    }
    return session.UpdatedAt > state.UpdatedAt
}

三种情况：首次发现（无本地记录）→ 同步；远端无更新时间（字段缺失）→ 保守同步（宁可多跑）；远端时间戳更新 → 同步。这是"保守优先"策略。

5.2 mergeSessionMetadata — 元数据合并

列表接口（ListSessions）和详情接口（GetSession）返回的 UserID/AgentID 可能不一致（API 设计问题）。mergeSessionMetadata() 以详情接口为主，列表接口为补充，最终兜底为 default_user/default_agent，确保路径生成的确定性。

5.3 updateSessionCursor — 游标只前进不后退

展开关键逻辑 
// 已有游标更新则保留，防止游标倒退
if ok && (prev.LatestMessageCreatedAt > latestCreatedAt ||
    (prev.LatestMessageCreatedAt == latestCreatedAt &&
     prev.LatestMessageID > latestID)) {
    latestID = prev.LatestMessageID
    latestCreatedAt = prev.LatestMessageCreatedAt
}

防止并发场景下不同轮次的游标互相覆盖导致倒退。采用"单调递增"语义：游标只能前进，不能后退。

5.4 retryBackoff — 指数退避 + Jitter

展开源码 
func retryBackoff(attempt int) time.Duration {
    base := 100 * time.Millisecond
    for i := 1; i < attempt; i++ {
        base *= 2 // 100ms → 200ms → 400ms
    }
    jitter := time.Duration(rand.Int63n(int64(base/2) + 1))
    return base + jitter
}

在 [0, base/2] 范围内加随机 jitter，打散多个实例的重试时间窗口，避免"惊群效应"（thundering herd）。

Section 7

7. 风险点分析

🔴 高风险：MaxMessageFetch 触顶导致消息永久遗漏

当两轮之间某 Session 新增消息数超过 MaxMessageFetch（默认 1000）时，fetchUntilBoundary 触顶返回 reachedMax=true，但仅记录 warn 日志，不会失败重试，也不会在下轮扩大窗口。超出范围的消息将被永久遗漏（无告警、无补偿）。

🔴 高风险：at-least-once 导致重复写入

进程在 WriteMessages 成功后、MarkMessagesWritten 执行前崩溃，重启后该批消息会再次写入 JSONL。JSONL 文件中可能存在重复行，下游消费方必须按 message_id 去重。

🟡 中风险：无 Schema 迁移机制

DB 结构变更需手动删除旧 DB 或指定新路径，无法平滑升级。生产环境中历史数据将全部丢失，重新从远端全量同步。对于大量 Session 的环境，重新同步可能耗时较长。

🟡 中风险：synchronous=NORMAL 掉电风险

SQLite WAL + NORMAL 同步模式在操作系统崩溃时可能丢失最近若干秒的已提交事务。虽然 at-least-once 保证重启后可重试，但状态可能短暂不一致。

🟢 低风险：locks map 内存增长（已有 TTL 清理）

FileSink 实现了 TTL 清理机制（10 分钟 TTL + 1 分钟清理间隔），长期运行的内存增长问题已得到缓解。但极端场景下（大量不活跃的 session，每个都有独立 JSONL 文件），map 仍可能累积较多条目。

🟢 低风险：无限制的 goroutine 数量（在 Worker Pool 内可控）

Worker 数量由 SessionWorkers（默认 8）控制，但每个 Worker 内的 HTTP 请求未做速率限制（rate limiting）。高并发场景下可能对 open-code 服务造成压力。

🟢 低风险：open-code API 无认证

当前 HTTPSource 不携带任何认证 Token，假设 open-code 在本地 localhost 运行。若未来需要接入远程服务或需要认证，需要在 HTTPSource 层增加 Authorization header 支持。

func (s *ESSink) OutputRoot() string { return s.index }
func (s *ESSink) PathFor(r domain.MessageRecord) string {
    return fmt.Sprintf("%s/%s/%s", s.index, r.SessionID, r.MessageID)
}

8.3 解决 MaxMessageFetch 遗漏问题

在 syncSession 中，当 reachedMax=true 时，可以在 Store 中记录"该 Session 需要深度补偿扫描"的标志，并在下次同步时使用不同策略（如基于游标分页而非 limit 窗口）回填历史消息。

8.4 引入 Schema 迁移

推荐引入 golang-migrate 或类似方案。迁移脚本按版本号顺序执行，确保平滑升级而不丢失历史数据。当前 checkSchema 可改为版本号比对。

8.5 增加认证支持

在 HTTPSource 添加 authToken string 字段，在 getOnce 中设置 Authorization: Bearer {token} header，并在 config.go 增加 -auth-token 参数。

8.6 扩展可观测性（Prometheus Metrics）

在 status.Reporter 基础上，增加 Prometheus 指标暴露（/metrics 端点），记录 rounds_total、messages_written_total、sessions_synced_total、max_fetch_reached_total 等计数器，便于接入 Grafana 监控看板。

8.7 at-least-once → exactly-once（Outbox 模式）

若需要 exactly-once 语义，可引入 Outbox 模式：

1
写 SQLite outbox将待写记录先原子写入 SQLite outbox 表（与状态更新在同一事务）
2
独立 outbox flusher单独 goroutine 读取 outbox，刷写到外部 Sink（JSONL/ES/S3）
3
写成功后删除 outboxSink 写成功后，在同一事务删除 outbox 记录并更新状态为 written

代价是增加了复杂度，且外部 Sink 仍需幂等性支持（如 ES 按 doc ID 覆盖写）。
